# Engineering Setup — 工程标配文件体系 v1

每个工程/项目必须包含以下 7 个标配文件（放在工程根目录）：

```
工程根目录/
├── ENTRY.md           ← 接入手则：新人/新Agent的第一站
├── work_log.md        ← 工作记录：有动作必记录
├── pitfalls.md        ← 踩坑记录：遇坑必记
├── tasks.md           ← 任务清单：条件具备直接执行
├── standards.md       ← 操作规范（按需）
├── report.md          ← 项目报告：进度/负责人/技术债务
└── README.md          ← 工程概述
```

## 文件模板

### 1. ENTRY.md — 接入手则

```markdown
# ENTRY.md — 接入守则

欢迎来到 **{工程名}**。

## 这是什么

{一句话描述}

## 你现在在哪

```
{工程根目录}
```

## 快速导航

| 你想知道 | 去看哪个文件 |
|----------|-------------|
| 🔄 进度到哪了、谁做了什么 | `work_log.md` |
| 📋 下一步要做什么 | `tasks.md` |
| 💥 踩过什么坑、怎么解决的 | `pitfalls.md` |
| 📊 项目全局评估 | `report.md` |
| 🏗️ 完整方案/设计文档 | `docs/{方案文档}.md` |
| 📐 操作规范/约定 | `standards.md` |
| 📖 工程目录结构 | `README.md` |

## 三条规矩

1. **先看后动** — 动手前先翻 `work_log.md` 和 `tasks.md`
2. **干了就记** — 每推进一步更新 `work_log.md`，遇坑记 `pitfalls.md`
3. **守规范** — 操作规范写在 `standards.md`，照着做

## 主要负责人

- **架构师/决策者**：{人名}
- **主力开发**：{人名}
- **辅助**：{人名}

## 需要帮忙？

- 项目问题 → 看 `tasks.md` + `pitfalls.md`
- 重大问题 → 找老板
```

### 2. work_log.md — 工作记录：**有动作必记录**

> **这是 agent 的「执行契约」，不是日志。**
> 你做完了一件事 = 你写下了一条记录。写完了才算做完。

#### 核心规则

```
有动作 → 写记录 → 更新 tasks → 下一步
   ↑_____________________________|
```

**做→记→更新→继续** 是 agent 的标准动作循环。

- ✅ 做完一个步骤 → 立刻写一条记录
- ❌ 不允许出现"做了事但没记"的情况
- ❌ 不允许"先做事，等晚点再补记录"
- **"我做了"的证据 = work_log 里有记录 + tasks 里打了勾**

#### 格式

```
{YYYY-MM-DD HH:MM} | {执行人} | {做了什么}
{一句话描述结果}
- ✅ / ⚠️ / ❌ {结果 + 关键数据}
- 关联 pitfalls: #{编号}（若有）
- 关联 tasks: #{编号}（若有）
```

#### 示例

```
2026-06-15 14:30 | Hermes | B-1 参数解析标准化
对 9544 条工作流提取 normalized_params，覆盖 4280 条（44.8%）
- ✅ 完成，耗时 19s，0 错误
```

```
2026-06-15 15:00 | Hermes | 修复索引偏移
发现节点类型 A 的 params_raw 索引与节点类型 B 不同，导致异常值
- ✅ 修复，0 异常值残留
```

## {YYYY-MM-DD} | {执行人} | {问题标题}

**解决**: {怎么修好的}

- {以后怎么做可以避免}

### 4. tasks.md — 任务清单

```markdown
# 任务清单 — Task List

> **条件具备直接执行，不等指令。**

### 任务 {编号}：{任务标题}

**状态**: ⏳ 待办 / 🔄 进行中 / ✅ 已完成 / ❌ 取消
**负责人**: {执行人}

**验收标准**:
- [ ] {条件}

**依赖**: 任务 {编号} 完成后才可开始
```

### 5. standards.md — 操作规范（按需创建）

```markdown
# 操作规范 — Standards

> 工程运行时必须遵守的规则。**有规范才建，没有就不建。**

## 路径规范

- 所有路径用 `Path(__file__).resolve().parent` 相对路径
- 不准硬编码绝对路径
- 启动脚本用 `SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"`

## 代码规范

| 规则 | 说明 |
|------|------|
| 禁止裸 `except:` | 全部用 `except Exception:` |
| 禁止 `eval()` | 用 `ast.literal_eval()` + 类型校验 |
| 函数不超过 200 行 | 超过则拆分 |

## 安全规范

| 规则 | 说明 |
|------|------|
| API Key 不写入代码 | 从环境变量或配置文件读取 |
| 外部输入不信任 | 所有用户输入做合法性校验 |
| 敏感文件不进 Git | 在 .gitignore 中声明 |

## 文档规范

- 每完成一个功能 → 更新 `work_log.md`
- 每踩一个坑 → 更新 `pitfalls.md`
- 任务完成后 → 更新 `tasks.md` 标记 ✅
- **记录是做完的一部分，缺记录不算完**
```

## 初始化步骤

```bash
# 1. 创建工程目录
mkdir -p {工程名}/data {工程名}/src {工程名}/scripts {工程名}/docs

# 2. 生成标配文件
touch {工程名}/ENTRY.md {工程名}/work_log.md {工程名}/tasks.md \
      {工程名}/pitfalls.md {工程名}/standards.md {工程名}/report.md \
      {工程名}/README.md

# 3. 逐文件填入模板，替换 {占位符}
```

## 核心规则

1. **先看 ENTRY.md** — 新 Agent/新人接入的第一站
2. **有动作必记录，不等指令** — 完成一件事后，**立刻**写 work_log。标准动作循环：做→记→更新 tasks→继续。不允许事后补记。
3. **遇坑必记 pitfalls.md** — 现象+根因+解决+预防，缺一不可。
4. **条件具备直接执行** — 不需要决策的事项不问，做了再记。
5. **记录是做完的一部分** — 完成标准 = 「活干了 + work_log 记录了 + tasks 更新了」。缺任何一项都不算完。
6. **README 只给概览** — 详细内容分散到其他文件
7. **report.md 定期更新** — 阶段性里程碑时更新
8. **工程文件放公共位置** — 所有相关 agent 都能访问。判断：其他 agent 需要看吗？需要→公共目录，不需要→个人工作区。

## 通用踩坑预防清单

### 文件操作
| 坑 | 预防 |
|----|------|
| 大 JSON 写入半截损坏 | compact 格式 + batch 写入 + 原子替换 |
| 多 Agent 并发写 | 写操作走队列，单一 agent 合并 |
| 恢复错旧备份 | 恢复前检查各备份状态，选最新的完整备份 |
| 工程文件放个人工作区 | 公共文件放公共目录，问：其他 agent 需要看吗？ |

### 代码
| 坑 | 预防 |
|----|------|
| 路径硬编码 | 所有路径用 `Path(__file__).resolve().parent` |
| 幂等检查遗漏 | 遍历数组精确匹配，不靠顶层字段 |
| 裸 `except:` | 全部用 `except Exception:` |
| `eval()` 安全风险 | 用 `ast.literal_eval()` + 类型校验 |
| 双向 import | 工具模块不引用业务模块 |

### 项目管理
| 坑 | 预防 |
|----|------|
| 活干了不记 | 完成标准 = 代码 + work_log + tasks 三项齐全 |
| 踩坑不记 | 遇坑立刻记 pitfalls |
| 等指令浪费时间 | 条件具备直接动手 |
| 新 Agent 接入迷茫 | ENTRY.md 就是第一站 |

### 跨 Agent 协作
| 坑 | 预防 |
|----|------|
| Agent 覆盖他人数据 | 写操作走队列，不直写公共数据 |
| 不知道其他人在做什么 | work_log 标注执行人，tasks 标注负责人 |
| 备份恢复搞混 | 备份命名加时间戳+状态标签 |
